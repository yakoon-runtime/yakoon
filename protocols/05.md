### Thema: Prompt-Flow & Batch
Unsere Engine muss 2 Dinge beherrschen:
1. Alles im Batch (ohne Prompt) "batch: delete 1; y; delete 2; y"
2. Prompt ohne Batch: delete 1; -> user prompt (Eingabe) -> y -> weiter 

Wenn innerhalb der while-loop der DialogManager blockiert, dann blockiert auch die while-loop.
Daher muss run_batch in einem task (create_task) laufen. Wir müssen dafür sorgen, dass jedes Blockieren
innerhalb des Batches nicht dafür sorgt, dass die While-Loop im Batch blockiert und dass der Context in 
die äußere Application-Loop fällt. Daher denke ich, dass wir auch innerhalb des Batches immer einen neuen 
Task starten müssten, wenn der vorherige blcokiert. 

Drei Bedingungen werden nun erfüllt:
1. Ein Command wird in die Engine gegeben. -> Prompt -> User macht Eingabe
    - command_inits = ["switch"]
2. Mehrere Commands werden im Batch abgearbeitet. KeyWord Batch:
    - command_inits = ["batch: login stefan; switch; mud; ic stefan; version;"]
3. Mehrere Commands werden im Batch abgearbeitet und die letzte Eingabe kann eine 
   Butzereingabe sein.
    - command_inits = ["batch: login stefan; switch; mud; ic stefan; version; switch;"]

### Thema: session dynamisches Routing
Es gibt Situationen, in denen zur Laufzeit für einen betimmten Moment Commands am Router registriert
werden müssen. Beispiel. Innerhalb eines Games steht der Spieler in einem Raum und kann durch Ausgänge mit n,s navigieren. Beide werden als Command registriert, und wieder entfernt, sobald der Raum 
verlassen wird. 

Ich denke wir brauchen ein Konzept, das weniger Tief eingreift. router.register_dynamics ist OK.
Bisher registriert ein char.move_to(room) die dynamsichen Commands im Router für dies Session. 
Das ist nicht gut. Besser wäre, wenn diese Logik direkt im Controller passiert. 
Soll kann beim Hook on_before_send() prüfen, ob eine session geladen und prüfen, 
ob es einen aktuellen Raum gibt character.location -> notfalls laden) 
Dann registrieren wir unsere Ausgänge (unseren router) kenne wir ja und erweitern dann doch 
die Session session.cmd_groups. on_after_send() entfernt alles dynamische wieder.

> Somit würde register + unregister im Controller stattfinden und nicht verteilt.
> durch die Erweiterung der cmd_groups bleibt die gesamte Logik im Resolver unverändert. Es 
  bedarf somit einer Spezialbehandlung. Notfalls könnten wir uns noch überlegen, 
  ob wie cmd_groups_dynamic einführen.

Ein Denkfehler: on_before_send wird nur dann ausgeführt, wenn ich ein gültiges Command eingebben 
habe. Das ist notwendig, dann nur dann steht die richtige Domain fest. Bedeutet: Ich habe keine 
Möglichkeit, im Vorfeld die Commands zu registrieren. Die einzige Möglichkeit die ich habe ist:

Alledings führt das dann dazu, dass wenn ein switch system kommt, dort ein ungültiges Command
eingegeben wird, dann on_before_send auf der letzten aktiven Domain ausgeführt wird.

Derzeit funktioniert das:
request -> register_dynamic -> command.run -> next request -> dynamic verfügbar
-> ABer das ist keine gute Lösung. Ich möchte dies
requst -> register_dynamic -> command_run -> dynamic verfügbar -> unregister_dynamic -> next request

Das Problem ist, dass ich den aktuellen Hook (domain) erst über das gültige Command erhalte. Commands
sind der Schlüssel, damit das System weiß, welcher Hook dafür zuständig ist. 
> Durch den switch ist immer eine Domain aktiv. Ich hätte nicht gedacht, das das am Ende so einfach wird.
> Durch diesen neuen Hook wird nun viels möglich, before das Command überhaupt zum run() kommt. 

Wir sollten einen Clean-up-Hook haben der immer läuft. Auch wenn Exceptions (Permissions) auftreten.

### Thema: loging an Plattform - und/nicht an Domain
Nachdem wir uns an der Plattform angemeldet haben, ist es manchmal notwendig, sich erneut anzumelden. Manchmal 
aber auch nicht. Das kann jede Domain selbst entscheiden. Dazu kann die Domain einen entsprechenden Text ausgeben, wenn das notwendig ist. Der Ort dafür könnte das Command switch sein. Switch gibt dann eine Willkommensmeldung aus und informiert, wie die Anmeldung erfolgen soll. Das Command Switch schaltet die domain dann in der Session auf die aktuelle domain. Dazu braucht es noch keinen Account.

**Problem**
- session.command_groups = account.groups wird durch den aktuellen Account überschrieben. command_groups
  wird an den verschiedensten Stellen geprüft. Diese werden bei jeder Session beim initialisieren durch
  die domains festgelegt. Somit legt jede Domain fest, was eine Session per Default tun darf 
  (Festlegung der commandsets.) Durch die Anmeldung (Account) kommen nun aber weitere groups hinzu. 
  Diese müssen der command_groups hinzugefügt werden, um nicht an allen Stellen im Code immer prüfen zu müssen:
  session.commandgroups + account. 

  Überlegung: Nachdem jede Session bei jedem Request neu aufgebaut, und der Account geladen wird (wenn account_id gesetzt), können wir die 
  account.cmd_groups direkt in die session übergeben und müssen diese nicht seperat an der Session verwalten.

  - Bedeutet: Nur der Account speichert die cmd_groups später im Storage. Die Session bekommt ihre gruppen druch die Domains + den Account 
    batch: login stefan; switch mud; help
